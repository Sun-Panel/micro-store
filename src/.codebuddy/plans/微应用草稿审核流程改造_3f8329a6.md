## 用户需求

修改微应用的创建和审核流程，实现草稿机制：

1. 创建微应用信息 = 创建草稿
2. 需要开发者点击"提交审核"才能进入审核流程
3. 审核通过才等于微应用正式发布成功
4. 每次修改后也是修改草稿，需要点击提交审核才能进入审核流程
5. 点击提交审核时需要检查必填项
6. 前后端都要配合修改
7. 减少冗余代码

## 产品概述

参考版本管理的实现模式（MicroAppVersion.status = -1 表示草稿），为微应用主信息增加草稿状态和独立的提交审核流程，使开发者可以自由创建和编辑微应用信息，准备就绪后再提交审核。

## 核心功能

- 草稿机制：创建微应用时状态为草稿，不进入审核
- 提交审核：新增独立的提交审核接口和前端入口
- 必填项校验：提交审核时校验必填项（应用名称、图标、分类等）
- 审核流程：审核通过后才正式上架发布
- 状态管理：清晰区分草稿、审核中、已上架、已拒绝等状态

## 技术方案

### 1. 状态字段调整

参考版本管理的成功模式（MicroAppVersion），调整微应用状态：

**MicroApp.status 调整为**：

- `-1`: 草稿（新增）
- `0`: 已下架
- `1`: 已上架
- `2`: 审核中

**reviewStatus 保持不变**：

- `0`: 无审核
- `1`: 审核中
- `2`: 已通过
- `3`: 已拒绝

### 2. 后端实现方案

#### 2.1 数据模型调整

**文件**: `service/models/microApp.go`

- 状态默认值从 `1` 改为 `-1`（草稿）

#### 2.2 业务逻辑调整

**文件**: `service/biz/micro_app_developer.go`

**调整 CreateApp 方法**：

- 创建时 status = -1（草稿），不再创建审核记录
- 创建时不设置 review_status

**新增 SubmitAppReview 方法**（提交审核）：

- 检查必填项：appName, appIcon, categoryId
- 检查是否已有待审核记录
- 创建审核快照记录（MicroAppReview）
- 设置 review_status = 1, status = 2

**调整 UpdateApp 方法**：

- 直接更新微应用信息，不创建审核记录
- 如果应用已上架，修改后保持上架状态（不受影响）
- 提供草稿保存功能

**保留 SubmitAppUpdate 方法**（但改名或重构）：

- 当前这个方法在更新时就创建审核记录，不符合新需求
- 建议重命名为 `SubmitAppReview`，专门用于提交审核

#### 2.3 API 接口调整

**文件**: `service/api/api_v1/admin/developer/micro_app.go`

**新增接口**：

```
POST /admin/developer/myMicroApp/submitReview - 提交审核
```

**调整现有接口**：

```
POST /admin/developer/myMicroApp/create - 创建草稿（status=-1）
POST /admin/developer/myMicroApp/update - 更新草稿（不进入审核）
```

#### 3.4 页面调整

- 头部根据状态显示不同操作按钮：

- 审核中：撤销审核、查看审核内容

- 提交审核时检查必填项，提示用户完善信息

### 4. 实现注意事项

**性能考虑**：

- 审核快照表（MicroAppReview）已有索引，无需额外优化
- 前端列表页已使用分页，性能无影响

**兼容性考虑**：

- 现有已上架应用不受影响（status=1）
- 现有审核中的应用，status=2 保持不变
- 只需在数据库迁移时，将现有的 status=2 且 review_status=0 的记录改为 status=-1

**错误处理**：

- 提交审核时，缺少必填项返回明确的错误提示
- 审核中状态不允许修改，需先撤销审核
- 已上架应用修改后仍保持上架，不影响用户使用

**日志记录**：

- 审核操作已有记录（MicroAppReview表）
- 无需额外日志
